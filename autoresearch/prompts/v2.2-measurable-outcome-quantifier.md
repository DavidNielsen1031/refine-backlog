# Prompt Variant v2.2 — Measurable Outcome Quantifier

## Experiment ID
exp_010 (proposed)

## Date
2026-03-11

## Hypothesis
In exp_009 (the most current real-world run with garbage detection), `has_measurable_outcome`
is the weakest **scored** dimension at 88.6% — with `has_testable_criteria`, `has_constraints`,
`no_vague_verbs`, and `has_verification_steps` all at or near 100%.

The 11.4% of specs failing `has_measurable_outcome` are specs where the refinement prompt
successfully adds verification steps, constraints, and acceptance criteria — but still produces
outcomes phrased as behaviors ("users can…", "the system will…") rather than observable/measurable
results ("returns HTTP 200", "latency < 200ms", "count increases by 1 in DB").

**Hypothesis:** The current gap guidance for `has_measurable_outcome` is too abstract. Adding
a concrete quantification micro-template — distinct from the acceptance criteria template — will
push pass-rate from 88.6% → 95%+.

## Root Cause (from exp_009 data)
Failing specs (has_measurable_outcome=false) cluster around two patterns:
1. Specs where the "outcome" sentence describes a feature capability but omits a metric
   (e.g., "users can filter results" rather than "filter returns results within 500ms")
2. Specs where `has_measurable_outcome` is false but `has_testable_criteria` is true —
   the criteria are specific but the outcome line stays vague.

## Proposed Change to rewrite-prompt.ts

In the gap guidance section, replace or augment the current `has_measurable_outcome` entry with:

```
- "has_measurable_outcome": The outcome must be **observable and quantified**. Add a
  "Success Metric:" line that states a concrete, testable result — not just what the feature
  does. Examples:
    • "Success Metric: API endpoint returns HTTP 200 with JSON body in < 300ms under load"
    • "Success Metric: Error count in Sentry drops to 0 within 24h of deploy"
    • "Success Metric: User can complete checkout in < 3 clicks, confirmed by QA walkthrough"
    • "Success Metric: Unit test coverage for this module increases from N% to ≥ 80%"
  Avoid: "users will be able to…", "the system should…" — these are behaviors, not metrics.
```

## Expected Impact
- has_measurable_outcome: 88.6% → 95%+
- has_testable_criteria: should remain at ~100% (not in conflict)
- avg_score: +1 to +2 points (each failing spec gains ~20 pts)
- agent_ready_pct: 98.9% → 99.5%+ (already near ceiling; marginal gain)

## Risk
- LOW: Quantification language could cause `has_testable_criteria` to double-count or conflict.
  Mitigation: the "Success Metric:" section is separate from "Acceptance Criteria:" — no overlap.
- LOW: Haiku may copy the template literally rather than adapting it. Mitigation: three diverse
  examples show different domains (API, observability, UX, test coverage).

## Measurement Criteria (pass/fail)
- PASS: has_measurable_outcome rises by ≥ 5 percentage points vs exp_009 baseline (88.6%)
- PASS: No regression on any other dimension (all remain ≥ 94%)
- FAIL: has_measurable_outcome stays flat or other dimensions drop by > 3%
