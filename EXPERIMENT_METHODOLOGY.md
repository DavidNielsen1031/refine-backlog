**Author:** David Nielsen, Perpetual Agility LLC

## Executive Summary

I ran 180 controlled AI coding agent experiments to test whether structurally rigorous specs produce better code from AI agents. The experiment compared three tiers of spec quality across 20 coding tasks, with 3 independent runs per combination.

**Result:** Vague specs (86.7% success) outperformed both hand-crafted detailed specs (76.7%) and Speclint-refined specs (75.0%). The primary failure mode was timeouts, not bad code — agents given more constraints spent more time trying to satisfy them.

---

## Research Question

> Does improving the structural quality of a backlog spec improve the output quality and completion rate of an AI coding agent implementing that spec?

### Hypothesis

Speclint-refined specs (with acceptance criteria, verification steps, assumptions, and priority) would produce higher completion rates and better code than vague or hand-crafted specs.

### Null Hypothesis

Spec quality has no statistically significant effect on AI agent coding success rates.

---

## Experimental Design

### Independent Variable: Spec Quality (3 Tiers)

**Tier A — Vague (Control)**
- Average ~16 words per spec
- Realistic "lazy developer" prompts — not intentionally bad, just minimal
- Example: *"Add pagination to the GET /api/items endpoint. Requirements: Support pagination. Return items in pages. Should be efficient."*

**Tier B — Detailed (Human-Crafted)**
- Average ~165 words per spec
- Written by a competent PM with specific acceptance criteria, error handling, and edge cases
- Example includes: cursor-based pagination, limit/cursor params with defaults and max values, nextCursor/hasMore response fields, sort order, error codes for invalid inputs

**Tier C — Speclint-Refined**
- Average ~275 words per spec
- Generated by running Tier A specs through the live Speclint API (speclint.ai/api/refine)
- Includes: structured problem statement, acceptance criteria with action verbs, assumptions, verification steps, estimate, priority with rationale, tags, and a success metric
- All scored 100/100 on Speclint's completeness rubric

### Dependent Variables

1. **Completion rate** — did the agent produce code that compiles (`tsc --noEmit`) AND passes all tests (`npm test`)?
2. **Failure mode** — success, timeout (10 min), or failure (broken code)
3. **Test count** — number of tests written by the agent (proxy for implementation depth)

### Controls

- **Same model:** Claude Sonnet 4 (anthropic/claude-sonnet-4-6) for all 180 runs
- **Same scaffold:** Identical Express.js + TypeScript project for every run
- **Same timeout:** 10 minutes per agent
- **Fresh environment:** Each run got a completely clean copy of the scaffold (separate directory, fresh node_modules via symlink)
- **No human intervention:** Agents ran autonomously from spec to completion
- **Same system prompt:** Identical instructions for every agent (see Appendix A)

## Execution Environment

### Agent Configuration
- **Model:** Claude Sonnet 4 (`anthropic/claude-sonnet-4-6`)
- **Runtime:** OpenClaw sub-agents via `sessions_spawn`
- **Timeout:** 10 minutes (600 seconds) per agent
- **Concurrency:** Up to 5 agents running simultaneously
- **Tools available:** `read`, `write`, `edit`, `exec` (file system + shell access)

### System Prompt (Appendix A)

Every agent received identical instructions:

```
You are a senior software engineer. You have been given an Express.js + TypeScript 
project scaffold with working dependencies (node_modules installed), a sample server, 
and passing tests.

Your job is to implement the feature described in the spec below. The project has:
- Express.js with TypeScript (strict mode)
- Jest + ts-jest for testing
- supertest for HTTP testing
- uuid, jsonwebtoken, crypto-js, stripe, multer available as dependencies

Instructions:
1. Read the existing src/server.ts to understand the codebase structure
2. Implement the feature described in the spec
3. Write comprehensive tests (in .test.ts files)
4. Ensure TypeScript compiles cleanly (tsc --noEmit)
5. Ensure all tests pass (npm test)

Do NOT modify package.json, tsconfig.json, or jest.config.js.
Do NOT install new packages.
Do NOT ask clarifying questions — make your best judgment on ambiguous requirements.
Write clean, production-quality TypeScript code.
```

### Success Criteria

A run was classified as:
- **Success:** TypeScript compiles (`tsc --noEmit` exits 0) AND all tests pass (`npm test` exits 0) AND at least one new test file was created
- **Timeout:** Agent did not complete within 10 minutes
- **Failed:** Agent completed but code doesn't compile or tests fail

## Key Findings

### 1. Vague Specs Won — Consistently

Tier A outperformed both Tier B and Tier C by 10+ percentage points. This held across categories and complexity levels, with the gap widening for more complex tasks.

### 2. Timeouts Were the Dominant Failure Mode

Out of 180 runs, only 2 produced genuinely broken code (1 in Tier A, 1 in Tier B). The other 35 failures were all timeouts. Agents given detailed specs spent more time:
- Parsing and re-reading specific acceptance criteria
- Trying to match exact error codes and response formats
- Debugging edge cases called out explicitly in the spec
- Fighting pre-existing scaffold code to satisfy constraints

### 3. The "Task 12 Effect" — Detail Kills Complex Tasks

Task 12 (cache middleware) was the starkest example:
- **Tier A (vague):** 3/3 success — agents built clean, simple caches
- **Tier B (detailed):** 1/3 success — 2 agents timed out fighting scaffold complexity
- **Tier C (Speclint):** 0/3 success — all 3 timed out trying to satisfy LRU eviction + TTL + invalidation + cache stats

The detailed spec didn't produce better caches. It produced agents that ran out of time building *exactly the cache described* instead of *a cache that works.*

### 4. When Agents Finish, They Almost Always Succeed

Success rate among non-timed-out runs: 143/145 = **98.6%**. The bottleneck isn't code quality — it's completion speed.

### 5. The Winning Formula: "What and Where, Not How"

"Add pagination to GET /api/items" (Tier A) tells the agent what to build and where. It doesn't specify cursor vs. offset, response format, or error codes — letting the agent pick the simplest approach for the codebase.

## Limitations and Caveats

### 1. 10-Minute Timeout May Over-Penalize Detailed Specs
With 30+ minutes, Tier B and C might have caught up. However, in real-world usage, speed matters — if your spec makes agents 2x slower for comparable output, that's a real cost.

### 2. Claude Sonnet 4 Is Exceptionally Good at Gap-Filling
This model infers sensible defaults from vague prompts. Weaker models (or models from 2 years ago) might genuinely benefit from more structure. Results may not generalize to GPT-3.5, Llama, or smaller models.

### 3. All Greenfield Express.js Scaffolds
Every task started from a clean scaffold. Bug fixes in large codebases, refactors, or migrations might tell a different story. Detailed specs might matter more when the context is complex and unfamiliar.

### 4. Success = Compiles + Tests Pass
I didn't measure code quality, maintainability, security, or semantic correctness. A vague spec that produces "working but wrong" code (correct behavior that doesn't match what a human wanted) is still a failure — I just wasn't measuring that failure mode.

### 5. Same Model for Generation and Execution
Speclint uses Claude Haiku to refine specs, and the coding agent uses Claude Sonnet. There may be an "impedance mismatch" between how one model structures specs and how another model interprets them.

### 6. No Statistical Significance Testing
With n=3 per cell, individual task results aren't statistically significant. The overall pattern across 180 runs is more meaningful, but a larger sample would strengthen confidence.

### 7. Scaffold Complexity Varied by Task
Later tasks (15-20) had more complex pre-existing scaffolds, which created additional friction for all tiers but disproportionately affected detailed specs that had to integrate with specific patterns.

## Reproduction Guide

### Prerequisites
- Node.js 20+
- Claude API access (anthropic/claude-sonnet-4-6)
- OpenClaw or any sub-agent orchestration system

### Steps

1. **Clone the repo:** `git clone https://github.com/DavidNielsen1031/speclint.git`
2. **Navigate to experiment:** `cd experiments/ab-agent-test-v2`
3. **Review tasks:** Each task in `tasks/task-NN/` has `tier-a.md`, `tier-b.md`, `tier-c.md`, and `meta.json`
4. **Review scaffold:** `scaffold/` contains the base Express.js + TypeScript project
5. **Run a single test:** Copy scaffold to a fresh directory, give an agent the system prompt (Appendix A) + one spec file, set a 10-minute timeout
6. **Judge results:** Check `tsc --noEmit` (exit code 0) and `npm test` (exit code 0, at least 1 new test file)

### Key Files
- `tasks/summary.json` — task distribution metadata
- `tasks/task-NN/meta.json` — per-task category and complexity
- `tasks/task-NN/tier-{a,b,c}.md` — the three spec tiers
- `scaffold/` — base project (copy per run)
- `scripts/generate-tasks.py` — task generation source
- `scripts/generate-tier-c.py` — Tier C generation via Speclint API
- `outputs/task-NN-{a,b,c}-run{1,2,3}/` — raw agent output per run

## Appendix B: Tier C Generation

Tier C specs were generated by sending each Tier A spec to the live Speclint API:

```python
POST https://speclint.ai/api/refine
Headers: { "x-license-key": "<SPECLINT_INTERNAL_KEY>" }
Body: { "items": ["<tier_a_spec_text>"] }
```

The API returned structured JSON with: title, problem statement, acceptance criteria, assumptions, verification steps, estimate, priority with rationale, and tags. All 20 specs scored 100/100 on Speclint's completeness rubric. One spec (task-08) returned a 500 error and was retried successfully.
