# Speclint Eval Audit Report

*Conducted: March 4, 2026*  
*Methodology: Hamel Husain's eval-audit skill (tools/evals-skills)*  
*Auditor: Alexander*

---

## Pipeline Summary

Speclint has **two evaluation layers:**

1. **LLM Refinement** (`/api/refine`) — Claude Haiku takes raw backlog items, produces structured JSON with title, problem, acceptance criteria, estimates, tags, assumptions
2. **Deterministic Scoring** (`lib/scoring.ts`) — Regex/heuristic rubric scores the LLM output on 5 dimensions (100pt scale), gates "agent_ready" at ≥70

**No traces stored. No human labels. No judge validation. No error analysis.**

---

## Findings (ordered by impact)

### 1. Error Analysis — MISSING (Critical)

**Status:** Problem exists

No systematic error analysis has been performed on Speclint outputs. We don't know:
- What kinds of specs does Speclint score wrong?
- When does the LLM generate bad acceptance criteria?
- What input patterns cause the scoring rubric to give false confidence?

Without error analysis, the scoring rubric was built from intuition, not observed failures.

**Fix:** Collect ~100 real API traces (we have ~17 Azure IPs hitting the API). Run `error-analysis` skill to categorize failure modes. This must happen before any judge/evaluator work.

### 2. Deterministic Scoring is Unvalidated (Critical)

**Status:** Problem exists

`scoring.ts` uses regex patterns to check 5 dimensions:
- `has_measurable_outcome` — regex for numbers/keywords in `problem` field (20pts)
- `has_testable_criteria` — regex for action verbs in ACs (25pts)
- `has_constraints` — tags ≥2 OR assumptions present (20pts)
- `no_vague_verbs` — title doesn't start with "improve/enhance/optimize" (20pts)
- `has_verification_steps` — regex for "verify/test/assert" in any text (15pts)

**Problems:**
- These dimensions were brainstormed, not observed from real failures
- Regex-based scoring is brittle — "The system returns 200" scores as testable, but "Make the button work" might also pass if it contains "the system"
- No human has rated whether a score of 75 actually means "agent-ready"
- `has_constraints` gives 20 points for having ≥2 tags — tags are generated by the LLM, not a quality signal
- `has_definition_of_done` is present in the breakdown but scores 0 points (dead code)

**Fix:** Label 100 specs with human Pass/Fail for "is this agent-ready?" Then measure whether the 70-point threshold aligns with human judgment. Use `validate-evaluator` skill.

### 3. No Stored Traces (High)

**Status:** Problem exists

Telemetry tracks metadata (tokens, cost, latency, scores) but the actual input specs and LLM outputs are not stored. The `trackUsage` call includes `items` (inputs) and `scores` (summary), but not the full refined output.

Without stored traces, we can't:
- Do error analysis retroactively
- Build labeled datasets
- Compare outputs across prompt versions

**Fix:** Add full trace storage — input spec, full LLM output, computed scores. Even a local JSONL file is sufficient to start. Redis/Upstash for production.

### 4. LLM Output Has No Quality Judge (High)

**Status:** Problem exists

The LLM generates acceptance criteria, problem statements, estimates, and priorities. The only validation is Zod schema conformance (`RefinedItemsSchema`). There's no check on whether the generated content is actually *good*.

Examples of undetected failures:
- ACs that are copy-paste of the input with no added specificity
- Problem statements that restate the title
- Estimates that are always "M" regardless of complexity
- Priorities that default to "HIGH" for everything

**Fix:** After error analysis identifies the top failure modes, build binary Pass/Fail judges for each using `write-judge-prompt`. Likely candidates: AC specificity, problem statement quality, estimate calibration.

### 5. Scoring Rubric Uses Vanity Dimensions (Medium)

**Status:** Problem exists

`has_constraints` rewards having ≥2 tags. Tags are LLM-generated ("bug", "feature", "ux") — their presence doesn't indicate spec quality. A terrible spec about "fix stuff" will still get tags from Claude.

Similarly, `has_measurable_outcome` triggers on any number in the problem field. "Fix the 3 bugs" scores as measurable even though it's not a measurable outcome.

**Fix:** Replace brainstormed dimensions with failure-mode-derived ones after error analysis. The scoring rubric should answer "what makes a spec fail in practice?" not "what sounds like a good spec attribute?"

### 6. No Pipeline Hygiene Process (Medium)

**Status:** Problem exists

No process to re-validate scoring after prompt changes, model switches, or feature additions. The scoring rubric hasn't been updated since initial creation. When persona scoring or discovery context was added, no re-evaluation happened.

**Fix:** Add eval re-run to sprint review checklist (see below). Any prompt or scoring change triggers a re-eval against the labeled dataset.

---

## What's OK

- **Schema validation** via Zod is solid — structural failures are caught and retried
- **Prompt injection monitoring** exists (non-blocking)
- **Telemetry** captures operational metrics well
- **Rate limiting and tier enforcement** is clean

---

## Recommended Sequence

```
1. Store traces          → SL-060 (prerequisite for everything)
2. Error analysis        → SL-061 (100 traces, categorize failures)  
3. Human labeling        → SL-062 (100 specs, binary agent-ready labels)
4. Validate scoring      → SL-063 (TPR/TNR of current rubric vs human labels)
5. Write judge prompts   → SL-064 (for top failure modes from error analysis)
6. Validate judges       → SL-065 (calibrate against human labels)
7. Sprint review gate    → SL-066 (re-run evals on every prompt/scoring change)
```

---

## Sprint Review Integration

Every sprint review that touches `/api/refine`, `scoring.ts`, or the refinement prompt **must include:**

1. **Eval re-run:** Run the labeled dataset through the pipeline, report TPR/TNR
2. **Regression check:** Compare current scores to baseline (no metric should drop >5%)
3. **New failure modes:** Any new patterns observed in traces since last sprint
4. **Dashboard update:** Post eval metrics to Speclint dashboard activity feed

This becomes a mandatory checklist item in the sprint review template.
